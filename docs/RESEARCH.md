# Holy Roof Rides — Research Report

## Landscape Summary: The Open-Source Ride-Sharing Ecosystem

Open-source ride-sharing has moved past "clone Uber" into a real architectural taxonomy, and the shape of that taxonomy matters for a project like Holy Roof Rides even though HRR sits at the small end of it. At one extreme is **Namma Yatri**, an India-deployed, protocol-compliant distributed system (Haskell/PureScript/Rust) built on the **Beckn Protocol** — an open network specification that splits ride-hailing into Beckn Application Platforms (BAPs, demand-side discovery/booking) and Beckn Provider Platforms (BPPs, supply-side fleets), talking to each other through signed, asynchronous request/callback pairs (`/discover` → `/on_discover`, `/confirm` → `/on_confirm`, etc.) brokered by **Beckn ONIX** middleware (Ed25519 signing, JSON schema validation, Go). This is infrastructure for federating *many* independent operators into one interoperable marketplace — solving a coordination problem Holy Roof Rides doesn't have, since it's one church, one trust boundary, one deployment.

At the other extreme is **Spatiad**, a single-binary Rust dispatch engine that holds all matching state — driver positions, jobs, offers — directly in memory with zero external database dependencies, using **Uber's H3** hexagonal spatial index to convert lat/lng into 64-bit cell IDs for O(1) proximity lookups instead of Haversine trigonometry or bounding-box queries. H3's hexagons are preferred over square grids because every neighboring cell is equidistant from center, removing directional bias from proximity search. Namma Yatri itself pairs its Haskell transactional core with a standalone Rust **Location Tracking Service** for exactly this reason — high-frequency GPS telemetry is cheap to handle in-memory/Redis but expensive to push through a relational database under load.

For routing, the landscape splits along a speed-vs-flexibility axis: **OSRM** precomputes Contraction Hierarchies over a road graph for sub-millisecond queries, but the whole graph must live in RAM and routing profiles are frozen at build time (no per-query toll avoidance or live traffic). **Valhalla** instead tiles the road network into small geographic files loaded on demand, trading some raw speed for dynamic per-query parameters (vehicle profiles, live weighting) and a far smaller memory footprint — better suited to constrained or edge deployments. Both consume OpenStreetMap data.

Finally, a cooperative/DePIN wing addresses ride-hailing's governance and commission problems outside the Beckn/Spatiad technical axis entirely. **CoopCycle** licenses its delivery-dispatch platform under a custom "Coopyleft" license that only permits commercial use by worker-owned cooperatives conforming to EU social-economy rules — a legal mechanism, not a technical one, to keep the commons out of extractive hands. Blockchain-based projects (DRIFE, DeRiS, RideChain, the now-defunct Teleport/TRIP Protocol) push driver reputation and payment escrow on-chain to avoid platform lock-in, with mixed real-world success — Teleport's 2025 shutdown illustrates that decentralization doesn't solve the "cold start" liquidity problem inherent to any two-sided marketplace. Closest in spirit to Holy Roof Rides is **OpenRide**, a GPLv3 self-hosted rider/driver/admin stack aimed at municipalities and community organizations running independent, non-commercial ride networks on their own fleets — validating the "small, self-hosted, non-marketplace" niche HRR occupies, even though (per the deeper prior-art research below) its specific tech stack is not one worth following.

## Prior Art: Community & Volunteer Ride Coordination

**OpenRide** (majordomo-studio/openride, GPLv3, ~6 stars/6 forks on GitHub) is a Vue.js + Vuetify ride-hailing proof-of-concept using Leaflet/OpenStreetMap for maps but Firebase + Airtable as its backend. The maintainers themselves flag it as dead: "no longer under development... Airtable should not be used in production as a back-end." Borrow: the OSM/Leaflet mapping choice validates HRR's own approach. Avoid: Airtable-as-database and the three-separate-web-app split (rider/driver/admin) — for a church-scale app a single mobile client with a role flag is simpler, and Firebase lock-in conflicts with HRR's self-hosted philosophy. Note there's naming confusion — several unrelated "openride" repos exist (openridesph/openrides, lcoenen/OpenRide, adam-nelson/openride) — worth double-checking which one any future collaborator means.

**HopHub** (HopHubProject/HopHub, AGPL-3.0, 25 stars/2 forks) is the most directly relevant prior art. Rails 8 + PostgreSQL + Redis/Valkey, Bootstrap/Haml frontend, deployed via Docker Compose. Its core trick: no user accounts at all — organizers create an event, share a link, and drivers/riders post offers/requests confirmed only by email; the system auto-matches and emails both parties to exchange contact info directly (no in-app chat or live tracking). Notable engineering details worth stealing: it proxies Geonames geocoding server-side so client IPs never leak to a third party, vendors all frontend assets locally (no CDN calls), runs a scheduled cleanup job that purges expired events/offers/requests, and ships a GDPR self-service tool (email lookup → data list + delete link). Borrow: the ephemeral-data-by-default posture (auto-expire) maps well onto HRR's "rides live in memory only" design — HopHub proves a whole product can be built around not persisting transactional data. Avoid: pure account-less design won't fit HRR since deacon-gated membership and PIN login are deliberate trust/safety requirements a church van ministry needs (accountability, repeat-rider history, safety reports) that ephemeral event links can't provide.

**Autofleet Locomotion** (Autofleet/locomotion, MIT, 50 stars/23 forks) is a React Native + Node.js rider app, but it's built to plug into Autofleet's commercial dispatch/fleet-management backend (payments, CRM, route optimization) — genuinely enterprise-grade, not a hobby project. Borrow: nothing structurally, though its RN component patterns for a request-a-ride flow could be skimmed. Avoid entirely as an architecture template — it assumes a paid backend platform and commercial-fleet complexity (surge logic, driver fleets, billing) that's the opposite of a single-congregation volunteer app; adopting it would mean stripping out most of what it does.

**LibreTaxi** (ro31337/libretaxi, AGPL-3.0, 3.9k stars/845 forks) is a Telegram-bot rideshare built in Go with PostgreSQL + RabbitMQ, launched 2016, peaked near tens of thousands of users, last released v0.1.0 in Nov 2017 and is now effectively unmaintained. It works "closer to Craigslist" — riders/drivers post into a shared channel and negotiate cash fares directly with zero platform trust or safety layer. Lesson for HRR: messenger-bot distribution (no app install) is real leverage for low-friction adoption, but LibreTaxi's total absence of vetting/moderation is precisely the failure mode HRR must not repeat — a church needs deacon approval and identity accountability that an open Telegram channel can't provide.

**Church transportation ministries today** mostly run on SignUp.com sheets, phone trees, and Facebook Events/Groups. Documented failure modes: coordinators spend ~15 hrs/week manually chasing confirmations; the same 20–25 reliable volunteers get re-called ("coordinator inertia") while willing others go inactive; friction in confirming (a phone call, an email reply) causes people to silently stop responding rather than decline; no-shows drop 30–50% when reminders are automated. Separately, church-owned van ministries carry real regulatory/liability weight: FMCSA requires registration for interstate faith-based transport even when exempt from full safety rules, $1.5M insurance is expected for vans seating ≤15, and documented driver training/MVR checks are the difference between a defensible position and "negligent entrustment" in litigation — relevant context for HRR's safety-reports table and any future driver-vetting feature, even though HRR itself is peer rideshare, not a chartered van service.

## OpenStreetMap Tiles for Mobile Apps: Policy, Alternatives, and Library Choice

**tile.openstreetmap.org policy is not safe for a distributed app.** The official OSMF Tile Usage Policy (operations.osmfoundation.org/policies/tiles/) is written for low-volume prototyping and single-user scripts, not for a published app with an unbounded number of installs. Key rules: (1) a unique, descriptive **User-Agent** naming the app plus contact info (e.g., `HolyRoofRides/0.1 (+contact@email)`) — no library-default UA, no impersonating a browser; (2) **client requests only tiles for the current viewport** — no prefetching, no "download for offline" — bulk/pre-seeded downloading is explicitly forbidden; (3) **honor server cache headers**, or cache ≥7 days if headers are unreadable; (4) visible **"© OpenStreetMap contributors"** attribution, not hidden behind toggles. There's no published numeric "heavy use" threshold (no fixed req/sec or req/day) — enforcement is informal: the OSMF ops team blocks IP ranges or User-Agents that degrade server performance, without warning. Because a congregation-sized but growing user base on a shared free resource is exactly the pattern that gets silently blocked, tile.openstreetmap.org is fine for local dev only, not for shipping in a TestFlight/APK build.

**Alternatives compared:**

- **OpenFreeMap** — genuinely free, no API key, no usage cap, no signup; funded by donations. Self-hostable (full stack is open-source, serves vector tiles via a Btrfs-image trick). Vector-only (MapLibre style JSON: Positron/Liberty/Dark/Fiord). Commercial use explicitly allowed. Biggest risk: it's a single enthusiast-run project with no SLA — good primary choice, weak alone for a production app people depend on for rides.
- **Protomaps** — PMTiles format, single static file, self-hostable on any S3-compatible storage (even a Raspberry Pi or the church's own box) for near-zero ongoing cost; can download the whole planet or a regional extract. Hosted API is free for noncommercial use; commercial use requires GitHub Sponsorship. Best long-term fit for a self-hosted-everything project like this one.
- **Stadia Maps** — 200,000 free "credits"/month but **free tier explicitly forbids commercial use**; paid starts at $20/mo (1M credits) once commercial. A church app soliciting/accepting any payment or donation link could trip the commercial clause — worth confirming before relying on it.
- **MapTiler** — free tier gives 100,000 "map loads"/month (older docs) or per current pricing 5,000 map sessions/month + 100k API requests; on free plan, service pauses (not billed) once exceeded. Raster or vector, easy raster fallback, no card required.
- **Carto** — free basemaps effectively restricted to CARTO enterprise/nonprofit grantees; the general free public tier caps around 75,000 mapviews/month with only Positron/Dark Matter styles and requires attribution linking carto.com. Not a great fit here — smaller free allowance and awkward licensing versus OpenFreeMap/Protomaps.

**Library choice for Expo SDK 54+:** `react-native-maps` `UrlTile` (raster) is simple and works with the native Google/Apple map view underneath, but as of SDK 54 the bundled version (1.20.1) does **not** support the newer config plugin — the plugin needs react-native-maps ≥1.22 and Expo SDK ≥53, so you may need to pin a newer version manually. It requires a custom **development build** (EAS Build or local prebuild) — not usable in plain Expo Go once native config is added. `@maplibre/maplibre-react-native` (vector, MapLibre Native) needs its own config plugin (adds a Podfile post_install hook on iOS, Android customizations) and **also requires a development build**, not Expo Go, plus real native compilation — heavier on a no-Mac Windows setup since iOS still needs EAS Build's cloud macOS runners either way.

**Recommendation:** For **v0.1**, ship with `react-native-maps` `UrlTile` pointed at **OpenFreeMap's raster-equivalent or Protomaps self-hosted PMTiles via a tiny raster proxy**, avoiding tile.openstreetmap.org entirely, with a clear custom User-Agent set server-side if proxying. This keeps the EAS dev-build requirement (already needed for GPS/live features) as the only native complexity, skips vector tooling. For **v0.2**, migrate to `@maplibre/maplibre-react-native` with self-hosted Protomaps PMTiles (extract just the congregation's metro region — a few hundred MB) for offline-friendly, cheap, unlimited vector maps under full self-host control, matching the project's own self-hosted-server ethos. Exact attribution required either way: visible **"© OpenStreetMap contributors"** text on the map (not hidden), plus **"© OpenFreeMap"** and **"© OpenMapTiles"** if using OpenFreeMap, or Protomaps' required attribution string if using their basemaps — check each provider's current attribution snippet at build time since wording can change.

## Shipping Holy Roof Rides from Windows (No Mac) — 2026 State of the Art

**iOS: EAS Build cloud + TestFlight is the correct default.** Expo's EAS Build free tier gives every account 15 Android and 15 iOS builds per month, 1 concurrent build, low queue priority, and a 45-minute build timeout — plenty for a small open-source church app doing periodic releases. `eas build --platform ios` compiles and code-signs the `.ipa` entirely on Expo's macOS cloud workers (no local Mac needed), and `eas credentials` can generate/manage the Apple Distribution certificate and provisioning profile interactively, or EAS can do it automatically on first build if you let it manage credentials. `eas submit --platform ios` then uploads the `.ipa` straight to App Store Connect/TestFlight using an App Store Connect API key (recommended) or Apple ID + app-specific password — no extra cost beyond the Apple fee. Paid EAS tiers exist if free-tier queueing becomes annoying: **Starter** is $19/mo ($45 build credit, high-priority queue, 2-hour timeout), **Production** is $199/mo ($225 build credit, 2 concurrent builds); usage beyond credit is billed per build-minute. For a low-volume church project, free tier is almost certainly sufficient.

**Apple Developer Program is mandatory and costs $99/year** (individual or organization) — required to get an Apple Distribution certificate, register the app's bundle ID, and use TestFlight, which allows up to 100 internal testers or 10,000 external testers via public link. There is no way around this fee for TestFlight distribution; it's an Apple requirement independent of Expo.

**Alternative: GitHub Actions macOS runner + fastlane** is viable but strictly more expensive/complex for this use case. macOS runner minutes are billed at a **10x multiplier** against the free 2,000 Linux-minutes/month quota on private repos (public repos get unlimited free Actions minutes, so this may be free if the repo is public), and overage after that is ~$0.062/min. You'd still need the same $99/year Apple Developer account, plus you'd hand-roll certificate/provisioning-profile management via `fastlane match` (needs its own encrypted git repo or secret storage) and `fastlane pilot upload`. This only makes sense if you want CI fully outside Expo's infrastructure or need custom native code EAS can't build — not the case here.

**Android: skip EAS Build for APK distribution — use GitHub Actions with `expo prebuild` + Gradle.** Since Holy Roof Rides distributes a sideloaded APK via GitHub Releases (not Google Play), you don't need Play App Signing or a Google Play Console account ($25 one-time) at all. A `ubuntu-latest` runner is free-tier friendly (1x minute multiplier, 2,000 free min/month on private repos, unlimited on public repos): checkout → setup Java/Node → `npx expo prebuild --platform android` → decode a base64 keystore secret into `android/app/` → `./gradlew assembleRelease` → upload the signed APK to a GitHub Release (`gh release upload` or `softprops/action-gh-release`). Generate the keystore once locally with `keytool -genkeypair` and store it plus `KEYSTORE_PASSWORD`, `KEY_ALIAS`, and `KEY_PASSWORD` as GitHub Actions repo secrets (base64-encode the `.jks`/`.keystore` file since secrets are text). Critically, keystore/signing setup must happen **after** `expo prebuild`, since prebuild regenerates `android/` and can overwrite `gradle.properties`.

**Recommended lowest-friction pipeline:** EAS Build + EAS Submit (cloud) for iOS/TestFlight; plain GitHub Actions `ubuntu-latest` + `expo prebuild` + Gradle for the signed Android APK attached to Releases. This avoids any macOS-runner minute costs, needs no fastlane/match setup, and stays entirely on free tiers except the unavoidable $99/year Apple fee.

**Accounts/secrets the maintainer must create:**
- Expo/EAS account (free) — for `eas build`/`eas submit`
- Apple Developer Program membership — $99/year
- App Store Connect API key (or Apple ID + app-specific password) — for `eas submit`, stored as EAS secret
- GitHub repo secrets: `ANDROID_KEYSTORE` (base64), `ANDROID_KEYSTORE_PASSWORD`, `ANDROID_KEY_ALIAS`, `ANDROID_KEY_PASSWORD`
- GitHub Personal Access Token / default `GITHUB_TOKEN` for the release-upload Actions step (usually automatic, no manual secret needed)
- No Google Play Console account needed (sideloaded APK, not Play Store distribution)

## Privacy & Safeguarding for a Church Rideshare App

**Data minimization: what ephemeral, in-memory ride/location design buys you — and what it doesn't.** Keeping rides and live GPS only in server RAM (never written to SQLite/disk) is a strong data-minimization move: there's no location database to breach, subpoena, or leak, and it satisfies the core privacy-by-design principle that data should be limited to what's operationally necessary and not retained "just in case." However, in-memory-only does *not* mean untraceable — location is still transmitted off-device to the server, so it exists on the wire and briefly in server process memory, is visible to anyone with server/process access while a ride is active, and could still be logged incidentally (app logs, crash reports, reverse proxies, hosting provider request logs) unless those are also scrubbed. It also doesn't address device-side exposure (OS location history, other apps) or protect against a compromised server capturing data live. True "ephemeral" claims for app-store purposes require that data not be retained beyond the immediate request and not be used to build any profile.

**Child safety/safeguarding — church best practice vs. app responsibility.** Common U.S. church "Safe Sanctuaries"-style policies (used across UMC, Baptist, Catholic, and nondenominational congregations) converge on: a **two-adult rule** (no solo adult-minor pairing in a vehicle unless immediate family, or an approved single-driver exception requiring written parental consent naming the driver, time, and destination); **driver minimums** of 21–25 years old; **background checks** including national/state criminal databases and the Dru Sjodin National Sex Offender registry, often required by the church's insurance carrier as a condition of coverage; and **MVR (motor vehicle record) checks** plus proof of current auto insurance for anyone driving minors. These are policy/process controls, not things software can verify — an app should *encode* structural guardrails (e.g., a "minor" ride-type flag, a hard block on single-driver/single-minor pairing that requires a deacon-recorded consent override, a driver-approval flag gated on the deacon confirming background-check/MVR completion offline) but should explicitly leave background-check adjudication, consent-form handling, and screening frequency to church policy — the app is not a compliance system of record.

**Volunteer driver liability/insurance basics.** A volunteer's personal auto policy is primary and usually extends to church errands, but coverage can be denied or capped if the insurer deems it "business use." Churches commonly add **Hired and Non-Owned Auto (HNOA)** coverage precisely to fill this gap, plus a liability umbrella — many carriers recommend ≥$1M per occurrence, with $2M umbrellas common; personal umbrella riders ($1–5M) run roughly $250–550/year for the first $1M. None of this is something the app enforces, but it should surface a one-time acknowledgment/waiver at driver signup pointing volunteers to their church's insurance requirements.

**App-store privacy declarations.** For an app storing only name/phone/PIN in SQLite and transmitting live GPS ephemerally: Apple's **App Privacy ("nutrition") label** would likely classify location as "Data Not Linked to You" *only if* identifiers are stripped before any storage and it's never linked back to identity — given rides are tied to a member account for coordination, it's more defensible to declare location as "Linked to You, Used for App Functionality, Not Used for Tracking." Google Play's **Data Safety form** has an explicit "ephemeral processing" carve-out: precise location transmitted off-device and used only in-memory for the immediate request, not retained and not used for ad/user profiling, must still be disclosed in the form but can be excluded from the public-facing safety section. Contact info (name/phone) should be declared as collected, linked to identity, not shared with third parties, used for account functionality.

### Practical Checklist (not legal advice)

- Confirm location truly never touches disk/logs (audit server logging, reverse-proxy/hosting logs, crash reporters) before claiming "ephemeral."
- Set explicit retention: SQLite rows for members/invites/safety-reports should have a defined deletion/export policy.
- Encode (don't just document) a two-adult / minor-ride safeguard and a deacon-recorded consent path.
- Require deacon sign-off gate before a driver account is marked "approved," referencing offline background-check/MVR completion.
- Point volunteer drivers to the church's insurance policy (HNOA + umbrella) at signup; don't let the app imply it provides coverage.
- File Apple's privacy label and Google's Data Safety form matching actual behavior, re-verify after any change to what's transmitted/stored.
- Publish a plain-language privacy note explaining the ephemeral-location design so leadership and parents can verify the claim.

## PIN-Based Auth, Invite Codes, and Session Security for a Self-Hosted Church Rideshare App

**PIN guessing math.** A 4-digit PIN has 10,000 possible values; 6-digit has 1,000,000. Without rate limiting, an attacker sending one guess/second exhausts a 4-digit space in ~2.8 hours (6-digit: ~11.5 days). With a hard cap of 5 failed attempts before a lockout, brute-forcing 10,000 combinations requires 2,000 lockout cycles — at even a light 15-minute lockout that's ~21 days of continuous attack against one account, and OWASP ASVS caps consecutive failed attempts at 100 regardless of design. OWASP's Authentication Cheat Sheet recommends the failure counter be tied to the **account**, not IP, since attackers rotate source IPs trivially. For comparison, real-world PIN-locking systems (Android: 30s lockout after 5 fails; iOS: wipe after 10) show throttling, not complexity, is what actually defeats online PIN guessing — a 4-digit PIN with real rate limiting resists online attacks reasonably well for a low-value trust boundary like a church invite app, even though it would be trivially weak offline.

**Lockout vs. fixed-window tradeoff.** A fixed window (e.g., "reset the fail counter every 60 minutes") is simple but gameable: an attacker can burst attempts right at the window boundary, getting two allowances close together. A true lockout/backoff (each failure increases delay, e.g. exponential: 1s, 2s, 4s... or a rolling lockout window) is harder to game but adds state and complexity. For a tiny self-hosted server with SQLite, a pragmatic middle ground is a rolling counter per account (not per fixed clock window) plus a monotonic lockout duration after N failures — cheap to implement, closes the boundary-gaming gap.

**scrypt parameters.** Current 2026 baseline guidance is N=2^17, r=8, p=1 (~128 MiB memory, per OWASP's Password Storage Cheat Sheet); for lighter interactive-login hardware, N=2^15 is acceptable, but N below 2^14 should be avoided. Argon2id is generally preferred where available, but scrypt is an accepted fallback — appropriate for a small self-hosted Node/Fastify box where memory-hard hashing plus rate limiting is the real defense (PIN entropy itself won't save it).

**Invite codes.** Best practice: generate with a CSPRNG at ≥32 bytes prior to encoding (or ~20+ random alphanumeric characters if kept short for human entry), store only a SHA-256 hash of the code server-side, set expiry (7–30 days is typical so leaked codes die on their own), enforce a max-uses cap, and support explicit revocation independent of expiry. Track inviter, timestamp, and redemption count for auditability — directly relevant since deacon approval is already a gate in this design.

**Session tokens.** OWASP requires ≥128 bits of entropy from a CSPRNG; 256-bit is a comfortable margin. Store only a hash (SHA-256 or HMAC-SHA-256) of the token server-side, never the raw value, and always set an expiry — critical since rides/GPS live only in server memory, so a stale, unexpired token is a bigger risk multiplier than usual.

**expo-secure-store.** It wraps iOS Keychain Services (hardware-backed, per-app encrypted) and Android Keystore (key material in the TEE, values in encrypted SharedPreferences). It protects against another app or a casual filesystem/backup extraction reading the raw PIN/session token, and can require device passcode/biometric re-auth. It does **not** protect against a jailbroken/rooted device, malware with root, or someone who has already unlocked the phone — device compromise defeats it.

**Where this design is strong:** small threat model (trusted congregation, not public internet targets), scrypt + rate limiting is proportionate to a 4–8 digit PIN, invite+deacon-approval prevents open signup, and memory-only ride/GPS data limits breach blast radius since there's no persisted location history to steal.

**Where it's weak:** no described per-device binding (a stolen/hashed session token works from anywhere), no mention of PIN retry backoff being progressive vs flat, and invite-code compromise (e.g., screenshot shared beyond intended recipient) has no described single-use option.

**Three cheap hardening upgrades:** (1) progressive PIN backoff (1s→2s→4s...capping at a lockout) instead of flat fixed-window counting; (2) deacon-initiated force-logout/PIN-reset (revoke all sessions + invalidate PIN) reachable from the existing approval UI, no new infra; (3) bind session tokens to a device-generated ID stored in expo-secure-store, checked on each request, so a leaked token alone is insufficient.

## Implications for Holy Roof Rides

**Trusted-community trust model vs. open marketplaces.** Nearly every architecture in the landscape summary — Beckn/ONIX, on-chain reputation (DRIFE, DeRiS, RideChain), CoopCycle's Coopyleft licensing — exists to solve trust and liquidity *between strangers across independent operators*. Holy Roof Rides has neither problem: it is one deployment for one congregation, and trust is established once, out-of-band, by a deacon. That makes invite-code + deacon-approval (already built) the right-sized substitute for signed credentials, on-chain reputation, or federated discovery — none of which HRR needs to build, evaluate, or maintain. The closer, more useful comparisons are HopHub (ephemeral event data, no accounts) and LibreTaxi (zero-trust Craigslist-style negotiation): HRR sits deliberately between them, keeping HopHub's "don't persist transactional data" discipline while rejecting its account-less model and LibreTaxi's total absence of vetting, because a church van ministry's real risk (an unvetted adult alone with a minor, or an unaccountable repeat no-show) is a people problem that only identity + deacon oversight — not clever anonymity — can address.

**Why in-memory beats H3/dispatch engines at congregation scale.** `server/src/state.js` already implements the same core insight as Spatiad — rides, sockets, and driver locations live only in a `Map()`, never touch SQLite, and evaporate on completion or restart — but without Spatiad's H3 hex-grid indexing or lock-free concurrent maps, and it doesn't need them. H3's entire value proposition is O(1) proximity lookup across a *large, dense* fleet where scanning every driver's raw coordinates would be too slow. A single congregation's active ride count is realistically single digits to low tens at any moment; `rides` is a `Map`, and `[...rides.values()]` with a plain distance comparison is faster to write, faster to review, and fast enough in absolute terms than standing up an H3 dependency, a hex-resolution tuning decision, and the geometry code to convert lat/lng at request time. The same logic rules out OSRM/Valhalla as dispatch-time components: HRR doesn't do bulk driver-matching over a routing graph, it broadcasts one rider's request to whichever drivers are in "Give a Ride" mode and lets a human pick. Routing/ETA display (if added later) is a client-side map concern, not a server dispatch concern — pulling in a routing engine would be solving a fleet-optimization problem HRR was explicitly scoped not to have.

**OSM tile choice.** The ROADMAP already commits to "OpenStreetMap tiles; no Google/Apple map dependency" and a Phase 2 move to MapLibre vector tiles — this matches the tile-policy research directly. For Phase 1 (`react-native-maps` `UrlTile`), HRR should not point at `tile.openstreetmap.org`: the OSMF policy's no-bulk-download and viewport-only rules are fine for a handful of testers but the informal enforcement (silent IP/UA blocking) is a real risk once the APK is distributed publicly per the roadmap's GitHub Actions release step. OpenFreeMap or a self-hosted Protomaps PMTiles extract (a single regional file the church's own $5 VPS can serve alongside the Fastify server) is the better Phase-1 default, with a custom `HolyRoofRides/x.y (+contact)` User-Agent set server-side if proxying, and visible OSM/OpenFreeMap/Protomaps attribution baked into the map screen from day one rather than retrofitted before a store submission. The Phase 2 MapLibre migration is exactly the OpenFreeMap/Protomaps vector path the research recommends, and self-hosting PMTiles is consistent with the project's "one small Node server + SQLite file... on a $5 VPS" self-hosting ethos already stated in the roadmap.

**EAS release path.** ROADMAP Phase 1 already lists "GitHub Actions: Android APK on every tag" and "iOS TestFlight path from Windows (EAS Build — no Mac required)" as shipped items — the research confirms this is the correct, lowest-cost pipeline: EAS Build's free tier (15 iOS/15 Android builds/month) plus `eas submit` covers TestFlight with no Mac and no cost beyond the mandatory $99/year Apple Developer fee, while Android's sideloaded-APK distribution needs no Play Console account at all and is cheaper to build via a plain `ubuntu-latest` GitHub Actions job (`expo prebuild` + Gradle + a keystore stored as repo secrets) than through EAS Build. One gap worth flagging against the roadmap: `react-native-maps` on Expo SDK 54 needs a pinned version ≥1.22 for the config plugin to work, which forces a development build (not Expo Go) — since GPS/live-ride features already require a dev build, this doesn't add new pipeline complexity, but the version pin should be called out explicitly in `app/` setup so a future contributor doesn't silently fall back to a broken default.

**Safeguarding checklist items for Phase 2.** The roadmap's Phase 2 list (push notifications, scheduled rides, ride notes & seat count, vector tiles, multi-congregation) has no explicit safeguarding line item yet — the research suggests adding one. Concretely: (1) a minor/seat-count-aware ride flag building on the planned "ride notes & seat count" feature, extended with a structural two-adult/single-driver-consent guard rather than leaving it to informal trust; (2) a deacon-reachable force-logout/session-revoke action, cheap to add to the existing admin approval UI and directly addressed by the PIN/session research's hardening recommendations; (3) a one-time driver-signup acknowledgment pointing volunteers to the church's own HNOA/umbrella insurance policy, since the app cannot and should not imply it provides coverage; (4) before any public store listing, an explicit audit confirming location data never lands in Fastify access logs, reverse-proxy logs, or a crash reporter, so the "ephemeral" claim in Apple's privacy label and Google's Data Safety form is actually true and not just architecturally intended. None of these require new infrastructure — they're additions to admin UI, signup flow, and logging configuration already in place.
